  <action>
Create build.js file in project root with the following implementation:

```javascript
#!/usr/bin/env node

import { execFileSync } from 'child_process';

// Read platform from environment variable, default to 'node'
const platform = process.env.DEPLOY_PLATFORM || 'node';

console.log(`\x1b[36m%s\x1b[0m`, `[build] Building for platform: ${platform}`);
console.log(`\x1b[90m%s\x1b[0m`, `[build] DEPLOY_PLATFORM: ${process.env.DEPLOY_PLATFORM || '(not set, using default: node)'}\n`);

// Execute Astro build using execFileSync for security
// This prevents shell injection vulnerabilities by separating command from arguments
const command = 'npx';
const args = ['astro', 'build'];

try {
  execFileSync(command, args, {
    stdio: 'inherit',
    env: { ...process.env, DEPLOY_PLATFORM: platform }
  });
  console.log(`\n\x1b[32m%s\x1b[0m`, `[build] ✓ Build completed for ${platform}`);
} catch (error) {
  console.error(`\n\x1b[31m%s\x1b[0m`, `[build] ✗ Build failed for ${platform}`);
  process.exit(1);
}
```

Security requirements:
1. Use execFileSync (NOT execSync) to prevent shell injection
2. Pass command and arguments as separate parameters (['astro', 'build'], not single string)
3. Inherit environment variables with { ...process.env } spread
4. Add DEPLOY_PLATFORM to env explicitly to ensure it's set

Logging requirements:
1. Log platform selection at build start (cyan color for visibility)
2. Log DEPLOY_PLATFORM environment value (gray color for debug info)
3. Log success/failure at build end (green/red colors)
4. Use ANSI color codes for terminal output (\x1b[36m for cyan, \x1b[90m for gray, \x1b[32m for green, \x1b[31m for red)

DO NOT:
- Use execSync with concatenated shell strings (security vulnerability)
- Use template literals for command building
- Use shell features (pipes, redirects)
- Log sensitive environment variables (only log DEPLOY_PLATFORM)
